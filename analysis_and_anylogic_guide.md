## Часть 1. Анализ ВКР (Отчёт Слудников)

### Общая оценка

ВКР посвящена разработке интеллектуальной системы классификации и сегментации рентгеновских снимков для ГБУЗ ПК «Чайковская ДГБ». Работа состоит из трёх глав:

1. **Глава 1** — предпроектный анализ деятельности больницы (отрасль, ресурсы, ИТ-инфраструктура, подсистемы).
2. **Глава 2** — анализ цифровой инфраструктуры, методы реинжиниринга, структура ИС.
3. **Глава 3** — разработка интеллектуальной системы: постановка задачи, математические модели нейросетей (CNN, U-Net), BPMN-модели «как есть» / «как будет», информационно-логическая модель БД, реализация (PyTorch + React/TypeScript).

Работа логически выстроена, содержит все необходимые разделы. Для практической работы по имитационному моделированию нужна **математическая модель процесса (СМО)**, которая описана в отдельном MD-файле.

## Часть 2. Анализ математической модели (math_model_vkr.md)

### 2.1 Описание модели

Модель описывает процесс рентгенологической диагностики как **систему массового обслуживания M/M/c** (формулы Эрланга-С).

Ключевые параметры:

| Параметр | Значение | Откуда |
|----------|----------|--------|
| N_год (годовой объём) | 6 800 исследований | Данные больницы за 2024 год |
| T_фонд (годовой рабочий фонд) | 1 600 часов | Данные больницы |
| c (врачи-рентгенологи) | 2 | Данные больницы |
| T_врач (время работы врача) | 12 мин (анализ 8 + заключение 4) | Данные больницы |
| T_полн (полный цикл) | 20 мин | Расчёт: 2+5+1+8+4 |

### 2.3 Выявленные проблемы

Несмотря на правильность арифметики, есть несколько **содержательных проблем**:

#### Проблема 1: Несоответствие N_год между ВКР и мат. моделью

В тексте ВКР (глава 1, раздел «Продукты (услуги) предприятия») указано:
> «Рентгенография (цифровой аппарат, **~3 000** исследований в год)»

В математической модели используется **6 800** исследований/год.

Разница в 2.3 раза — это критическое расхождение. Если реальное значение 3 000, то:
- λ = 3000/1600 = **1.875** исслед./час (а не 4.25)
- ρ = 1.875/(2·5) = **0.1875** (а не 0.425)
- Система и без ИИ работает с огромным запасом, и эффект от ИИ будет менее впечатляющим

**Рекомендация:** Привести цифры к единому значению. Если 6 800 — это все инструментальные исследования (рентген + УЗИ + ЭКГ), а 3 000 — только рентген, это нужно явно пояснить в тексте. Для модели рентгенологической диагностики корректно использовать только рентгеновские исследования.

#### Проблема 2: «Удвоение пропускной способности» — некорректная трактовка

В модели утверждается:
> «Пропускная способность: 34 → 68 исследований за смену (+100%)»

Это **потенциальная** пропускная способность (сколько врачи *могут* обработать), а не фактическая. Если поток пациентов (λ) не изменился, то за смену по-прежнему поступает λ·8 = 4.25·8 = **34 пациента**. Просто врачи будут больше простаивать.

**Рекомендация:** Переформулировать: «При сохранении текущей загрузки врачей (ρ=0.425) потенциальный объём обрабатываемых исследований может быть увеличен до 68 за смену (+100%), что создаёт резерв для обработки будущего роста нагрузки».

#### Проблема 3: K_ρ штрафует за «слишком хорошую» эффективность

Формула K_ρ = 1 − |ρ_AI − 0.75| при ρ_AI = 0.2125 даёт K_ρ = 0.4625.

Это означает, что модель считает загрузку врачей в 21% **неоптимальной** (оптимум — 75%). Формально это верно с точки зрения теории СМО (недозагрузка ресурсов), но в контексте ВКР это может вызвать вопросы на защите: «Зачем внедрять систему, если врачи после этого будут простаивать 80% времени?»

**Рекомендация:** Дополнить интерпретацию: высвобожденное время врачей может быть использовано для консультаций, работы с телемедицинскими запросами, повышения квалификации и т.д. Либо рассмотреть сценарий с сокращением до 1 врача (c=1), что при ρ' = 4.25/10 = 0.425 — такая же загрузка, как исходный сценарий с 2 врачами.

#### Проблема 4: Предположение о независимости ошибок

Модель качества (раздел 5) предполагает **независимость ошибок** врача и ИИ:
> P_совм = 1 − (1−P_H)(1−P_AI) = 0.9805

Это сильное допущение. На практике ошибки врача и ИИ могут быть **коррелированы**: оба могут пропустить одну и ту же малозаметную патологию. Однако для учебной работы это приемлемое упрощение.

**Рекомендация:** Добавить примечание: «Расчёт выполнен в предположении независимости ошибок, что является верхней оценкой. На практике совместная точность может быть несколько ниже из-за корреляции ошибок на сложных случаях».

### 2.4 Улучшенная версия математической модели

Ниже представлена скорректированная версия с устранением выявленных проблем. **Изменения минимальны** — арифметика верна, корректируется только интерпретация.

#### Ключевые правки:

1. **Уточнение N_год:** Если используем 6 800, то пояснить, что это общий объём исследований кабинета лучевой диагностики (рентген + повторные + контрольные), а не только первичная рентгенография.

2. **Пропускная способность:** Заменить «удвоение фактической пропускной способности» на «удвоение потенциальной пропускной способности».

3. **Целевая функция — добавить ограничение:**

   F = α·ΔW + β·ΔN + γ·K_ρ → max

   **при ограничениях:**
   - ρ < 1 (устойчивость системы)
   - W_цикл ≤ 15 мин (целевое время диагностики)
   - P_miss ≤ 1% (допустимая доля пропущенных патологий)

   Эти ограничения делают модель строже и убедительнее для защиты.

4. **K_ρ — альтернативная формулировка:**

   Можно заменить на показатель «запаса мощности», который не штрафует за низкую загрузку:

   K_reserve = 1 − ρ_AI = 1 − 0.2125 = 0.7875

   Это показывает, что 78.75% мощности остаётся в резерве для пиковых нагрузок.

5. **Добавить анализ чувствительности:** Что произойдёт, если λ вырастет на 20%, 50%, 100%?

| λ, исслед./час | ρ (без ИИ) | W_цикл (без ИИ) | ρ (с ИИ) | W_цикл (с ИИ) |
|---|---|---|---|---|
| 4.25 (текущий) | 0.425 | 22.65 мин | 0.213 | 14.36 мин |
| 5.1 (+20%) | 0.510 | 23.4 мин | 0.255 | 14.5 мин |
| 6.4 (+50%) | 0.640 | 25.8 мин | 0.320 | 14.8 мин |
| 8.5 (+100%) | 0.850 | 35.6 мин | 0.425 | 15.5 мин |

Это показывает: без ИИ система с трудом справляется при удвоении нагрузки (ρ=0.85, время >35 мин), а с ИИ — работает комфортно (ρ=0.425, время 15.5 мин).

## Часть 3. Подробная инструкция по AnyLogic

### 3.1 Что такое AnyLogic (для новичка)

AnyLogic — это среда имитационного моделирования, где ты строишь модель из готовых блоков и запускаешь «виртуальную симуляцию» реального процесса. Это как компьютерная игра, где ты создаёшь виртуальную больницу и смотришь, как через неё проходят пациенты.

Для нашей задачи мы будем использовать **дискретно-событийное моделирование (DES)** — это когда объекты (направления на рентген) движутся по цепочке блоков (регистрация → рентгенография → анализ → заключение).

### 3.2 Установка и первый запуск

1. **Скачай AnyLogic:** Перейди на https://www.anylogic.com/downloads/ → скачай **AnyLogic Personal Learning Edition** (PLE) — бесплатная версия для учёбы.
2. **Установи:** Запусти установщик, следуй инструкциям. Требуется Java (обычно ставится автоматически).
3. **Запусти AnyLogic**, создай новый проект: `File → New → Model`.

### 3.3 Создание модели: Пошаговая инструкция

#### Шаг 1. Создание нового проекта

1. Открой AnyLogic.
2. `File → New → Model`.
3. Имя модели: `XRayDiagnostics`.
4. Нажми `Next`, выбери `Discrete Event` (дискретно-событийное моделирование).
5. Нажми `Finish`.

У тебя откроется пустой агент `Main` — это «холст», на котором мы будем строить модель.

#### Шаг 2. Определение параметров модели

Нам нужно создать параметры, которые можно будет менять между экспериментами.

В палитре слева найди `Agent` → перетащи элемент **Parameter** на холст `Main`. Создай следующие параметры:

| Имя параметра | Тип | Значение по умолчанию | Описание |
|---|---|---|---|
| `arrivalRate` | double | 4.25 | Интенсивность потока (исслед./час) |
| `numDoctors` | int | 2 | Количество врачей-рентгенологов |
| `regTime` | double | 2 | Время регистрации (мин) |
| `xrayTime` | double | 5 | Время рентгенографии (мин) |
| `pacsTime` | double | 1 | Время загрузки в PACS (мин) |
| `analysisTime` | double | 8 | Время анализа врачом (мин) |
| `reportTime` | double | 4 | Время формирования заключения (мин) |
| `aiEnabled` | boolean | false | Включён ли ИИ-сценарий |
| `aiAnalysisTime` | double | 4 | Время анализа с ИИ (мин) |
| `aiReportTime` | double | 2 | Время заключения с ИИ (мин) |

Для каждого параметра:
1. Перетащи элемент `Parameter` из палитры `Agent` на холст.
2. В свойствах (справа) задай **Name**, **Type**, **Default value**.

#### Шаг 3. Построение процессной цепочки (Flowchart)

Открой палитру **Process Modeling Library** (слева). Перетаскивай блоки на холст и соединяй их стрелками (клик на выходной порт → тянешь к входному).

Схема процесса:

```
[Source] → [Delay: Регистрация] → [Queue: Очередь на рентген] → 
[Seize: Рентген-аппарат] → [Delay: Рентгенография] → [Release: Рентген-аппарат] →
[Delay: Загрузка PACS] → [Queue: Очередь к врачу] → 
[Seize: Врач] → [Delay: Анализ] → [Delay: Заключение] → [Release: Врач] → [Sink]
```

Создай блоки по порядку:

**Блок 1: Source (Источник направлений)**
- Перетащи `Source` из палитры на холст.
- Name: `sourceReferrals`
- Arrivals defined by: `Rate`
- Arrival rate: `arrivalRate` (ссылка на параметр)
- Единицы: `per hour`

**Блок 2: Delay (Регистрация)**
- Name: `delayRegistration`
- Delay time: `triangular(1, regTime, 3)` (треугольное распределение 1–2–3 мин)
- Capacity: `1` (одно окно регистрации)

**Блок 3: Queue (Очередь на рентген)**
- Name: `queueXray`
- Capacity: `100` (максимум 100 человек в очереди)

**Блок 4: Seize (Захват рентген-аппарата)**
- Name: `seizeXray`
- Resource pool: `xrayPool` (создадим позже)
- Number of units: `1`

**Блок 5: Delay (Рентгенография)**
- Name: `delayXray`
- Delay time: `triangular(3, xrayTime, 8)` (треугольное распределение 3–5–8 мин)

**Блок 6: Release (Освобождение рентген-аппарата)**
- Name: `releaseXray`
- Resource pool: `xrayPool`

**Блок 7: Delay (Загрузка в PACS)**
- Name: `delayPacs`
- Delay time: `pacsTime` (фиксированное, 1 мин)

**Блок 8: Queue (Очередь к врачу)**
- Name: `queueDoctor`
- Capacity: `100`

**Блок 9: Seize (Захват врача)**
- Name: `seizeDoctor`
- Resource pool: `doctorPool` (создадим позже)
- Number of units: `1`

**Блок 10: Delay (Анализ снимка)**
- Name: `delayAnalysis`
- Delay time: (см. настройку с условием ниже)
  ```java
  aiEnabled ? exponential(1.0 / aiAnalysisTime) : exponential(1.0 / analysisTime)
  ```
  **Пояснение:** `exponential(1.0/mean)` — экспоненциальное распределение со средним `mean` минут. Если ИИ включён, среднее 4 мин, иначе 8 мин.

**Блок 11: Delay (Формирование заключения)**
- Name: `delayReport`
- Delay time:
  ```java
  aiEnabled ? normal(aiReportTime, 0.5) : normal(reportTime, 1)
  ```
  **Пояснение:** Нормальное распределение — со средним 2 мин (с ИИ) или 4 мин (без ИИ) и стандартным отклонением.

**Блок 12: Release (Освобождение врача)**
- Name: `releaseDoctor`
- Resource pool: `doctorPool`

**Блок 13: Sink (Завершение)**
- Name: `sinkCompleted`

#### Шаг 4. Соединение блоков

Наведи мышь на **выходной порт** блока (маленький треугольник справа) → зажми левую кнопку мыши → тяни к **входному порту** следующего блока (слева). Соедини все 13 блоков последовательно.

#### Шаг 5. Создание пулов ресурсов (ResourcePool)

Перетащи из палитры два элемента **ResourcePool**:

**Пул 1: Рентген-аппараты**
- Name: `xrayPool`
- Capacity: `2` (2 рентген-аппарата)

**Пул 2: Врачи-рентгенологи**
- Name: `doctorPool`
- Capacity: `numDoctors` (ссылка на параметр — по умолчанию 2)

#### Шаг 6. Добавление измерений времени (TimeMeasure)

Для измерения KPI нужны два элемента **TimeMeasureStart** и **TimeMeasureEnd**:

**TimeMeasureStart:**
- Name: `tmStart`
- Расположи **между** блоком Source и первым Delay (регистрация). Для этого: вставь `tmStart` в цепочку — входной порт подключи к выходу Source, выходной порт к входу delayRegistration.

**TimeMeasureEnd:**
- Name: `tmEnd`
- Расположи **перед** Sink — между releaseDoctor и sinkCompleted.
- В свойствах: `Time measure start`: выбери `tmStart`.

Теперь `tmEnd` будет автоматически считать время от входа в систему до выхода (время цикла).

#### Шаг 7. Добавление статистики и графиков

Для сбора и визуализации KPI перетащи с палитры **Analysis**:

1. **HistogramData** — для гистограммы времени цикла:
   - Name: `histCycleTime`

2. **TimePlot** — для отслеживания загрузки врачей во времени:
   - Name: `plotUtilization`
   - Добавь data item: `doctorPool.utilization()`

3. **BarChart** — для текущего состояния очередей:
   - Name: `barQueues`
   - Добавь data items: `queueXray.size()`, `queueDoctor.size()`

Для записи времени цикла в гистограмму: в блоке `tmEnd` найди поле **On measure**: впиши код:
```java
histCycleTime.add(ds.time);
```

Также добавь переменные для отслеживания:

| Имя переменной | Тип | Начальное значение | Описание |
|---|---|---|---|
| `totalCompleted` | int | 0 | Счётчик завершённых исследований |
| `totalCycleTime` | double | 0 | Суммарное время цикла |

В блоке `sinkCompleted` → поле `On enter`:
```java
totalCompleted++;
```

В блоке `tmEnd` → поле `On measure`:
```java
histCycleTime.add(ds.time);
totalCycleTime += ds.time;
```

#### Шаг 8. Настройка времени моделирования

В свойствах модели (`Main` → Properties):
- **Stop time**: `540` минут (8 часов = 480 мин смена + 60 мин прогрев)
- **Time units**: `Minutes`

Для периода прогрева: добавь событие (Event) на 60-ю минуту, которое обнуляет статистику:
1. Перетащи элемент **Event** из палитры `Agent`.
2. Name: `warmupEvent`
3. Trigger type: `Timeout`
4. Timeout: `60` (минут)
5. Action:
```java
totalCompleted = 0;
totalCycleTime = 0;
histCycleTime.reset();
```

### 3.5 Автоматизация экспериментов (Parameter Variation)

AnyLogic позволяет автоматически прогнать множество вариантов:

1. В дереве проекта (слева) щёлкни правой кнопкой на модель → `New → Experiment`.
2. Тип: **Parameter Variation**.
3. Name: `ParameterVariation`.
4. В настройках:
   - **Parameters**: добавь `numDoctors` с диапазоном 1–3 (шаг 1).
   - **Replications**: 10 (повторы для статистики).
   - **Stop condition**: модельное время 540 минут.

Для Эксперимента 2 создай второй эксперимент типа Parameter Variation:
- Варьируемый параметр: `aiEnabled` (false, true).
- Replications: 10.

### 3.6 KPI — Что и где смотреть

После запуска модели ты увидишь анимацию процесса. Для получения числовых результатов:

| KPI | Где смотреть | Формула/элемент |
|-----|------------|-----------------|
| Среднее время цикла | tmEnd → среднее | `totalCycleTime / totalCompleted` |
| Пропускная способность | Переменная | `totalCompleted` (за 480 мин после прогрева) |
| Загрузка врачей | ResourcePool | `doctorPool.utilization()` — значение от 0 до 1 |

| Средняя длина очереди к врачу | Queue | `queueDoctor.statsSize.mean()` |
| Максимальная длина очереди | Queue | `queueDoctor.statsSize.max()` |

## Часть 4. Связь математической и имитационной моделей

Это важный раздел отчёта. Вот таблица соответствия:

| Параметр мат. модели | Значение | Элемент AnyLogic | Настройка |
|---|---|---|---|
| λ = 4.25 исслед./час | Пуассоновский поток | Source (`sourceReferrals`) | Rate = `arrivalRate` per hour |
| μ = 5 исслед./час (12 мин) | Экспоненциальное | Delay (`delayAnalysis` + `delayReport`) | exponential(1.0/analysisTime) + normal(reportTime,1) |
| μ' = 10 исслед./час (6 мин) | Экспоненциальное | Те же Delay'и | exponential(1.0/aiAnalysisTime) + normal(aiReportTime,0.5) |
| c = 2 | Каналы обслуживания | ResourcePool (`doctorPool`) | Capacity = `numDoctors` |
| ρ = λ/(cμ) | Коэффициент загрузки | Статистика ResourcePool | `doctorPool.utilization()` |
| W_q | Время ожидания в очереди | Queue (`queueDoctor`) | `queueDoctor.statsWaitTime.mean()` |
| W_цикл | Полное время цикла | TimeMeasure | `tmEnd → mean` |

**Дополнительные элементы имитации, не учтённые в мат. модели:**

- Прогрев модели (60 мин).
- Конечная длительность смены (480 мин).

## Часть 5. Концептуальная имитационная модель

### 5.1 Сущности

| Сущность | Атрибуты | Роль |
|---|---|---|
| Направление (Referral) | ID, время поступления, приоритет | Заявка, движущаяся по системе |
| Врач-рентгенолог (Doctor) | ID, состояние (свободен/занят) | Ресурс обслуживания |
| Рентген-аппарат (XRayMachine) | ID, состояние | Ресурс |

### 5.2 Процессы

```
Направление поступает → Регистрация (2 мин) → Ожидание рентген-аппарата →
Рентгенография (3–8 мин) → Загрузка в PACS (1 мин) →
[если ИИ: автоанализ 5 сек] → Ожидание врача →
Анализ врачом (8 мин / 4 мин с ИИ) → Заключение (4 мин / 2 мин с ИИ) →
Исследование завершено
```

### 5.3 Логика очередей

- Очередь на рентген: **FIFO** (первый пришёл — первый обслужен).
- Очередь к врачу: **FIFO** (в базовом сценарии). Можно добавить приоритет для патологий, выявленных ИИ (расширение).

### 5.4 Целевая функция

**Минимизировать** среднее время цикла диагностики W_цикл:

$$W_{цикл} → min$$

**При ограничениях:**
- ρ ≤ 0.85 (загрузка врачей не должна превышать 85%)
- W_цикл ≤ 20 мин (целевое время)
- Число врачей c ∈ {1, 2, 3}

## Часть 6. Частые ошибки в AnyLogic (и как их избежать)

| Ошибка | Симптом | Решение |
|--------|---------|---------|
| Не подключён ResourcePool к Seize/Release | Ошибка при запуске «Resource pool not specified» | В свойствах Seize/Release выбрать ResourcePool из выпадающего списка |
| Время в неправильных единицах | Результаты в 1000 раз больше/меньше ожидаемого | Проверить, что модель работает в минутах (Properties → Time units) |
| Нет прогрева | Первые результаты завышены из-за пустой системы | Добавить Event с обнулением статистики на 60-й минуте |
| Очередь переполнена | Объекты исчезают | Увеличить Capacity очереди или проверить параметры |
| Source генерирует слишком много | Система перегружена | Проверить единицы: arrivalRate должен быть «per hour», не «per minute» |
